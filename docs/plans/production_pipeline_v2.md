# Music Lab Production Pipeline v2.0 — End-to-End 자동화

> **작성일**: 2026-04-23
> **작성자**: PM (orchestrator) · 구현 주체: music-lab 세션
> **대상 범위**: 컨셉 시드 → 완성곡 → YouTube 게시 → 피드백 수집 전 단계 자동화
> **원칙**: Boil the Lake (완전성) + 병렬화 최대화 + 재작업 최소화

---

## 0. 왜 v2인가 — v1의 한계

| 현 상태 (v1) | 문제 |
|-------------|------|
| `suno_pipeline.py` = 생성+다운+Drive만 | 컨셉→가사→프롬프트 단계가 분절 (사람 수동) |
| 품질 선별 = 사용자가 직접 들어야 | **픽** 자동화·알림·비교 리포트 없음 |
| 후처리 분리 (Demucs/Pedalboard/Matchering 스크립트 산재) | 파이프라인 안에 안 엮임 |
| Drive 서비스 계정 JSON 손상 (2026-04-23 확인) | 자동 업로드 불가 |
| YouTube 업로드는 별도 스크립트 `youtube_upload*.py` | 메타데이터 수동 작성 |
| 앨범/EP 단위 묶음 관리 없음 | 일관성·순서·크로스페이드 수동 |
| 피드백 루프 없음 | 어떤 곡이 왜 잘됐는지 학습 0 |
| 병렬화 부재 | 3시안을 시안별로 순차 실행 → 총 시간 n배 |

**v2의 약속**: 사용자가 "컨셉 시드 한 줄 + 비토/픽" 만 개입. 나머지는 파이프라인이 병렬로.

## 2. 병렬화 요약 — 시간 단축 효과

| 단계 | 직렬 시간 | 병렬 시간 | 단축 |
|------|----------|----------|------|
| P2 가사 3시안 | 90s × 3 = 270s | 90s | **3×** |
| P5 Suno 생성 | 3분 × 3 = 9분 | 3분 | **3×** |
| P6 다운로드 | 15s × 6 = 90s | 20s | **4.5×** |
| P7 품질 분석 | 10s × 6 = 60s | 15s | **4×** |
| P9 stems 후처리 | 30s × 4 = 120s | 40s | **3×** |
| P10 아트워크 | 45s × 3 = 135s | 50s | **2.7×** |
| P11 메타데이터 | 30s × 3 = 90s | 35s | **2.6×** |
| P12 배포 | 120s × 3 = 360s | 130s | **2.8×** |

**총 end-to-end 직렬 시간 추정**: ~35~45분
**총 end-to-end 병렬 시간 추정**: **~12~15분**
**사용자 개입**: ◆ 3지점 (시드 / 톤 방향 / 픽 + 메타 승인) = 5분 내

## 3. 신규 기능 상세 (우선순위 순)

### ★ F-01. 품질 자동 분석 + 추천 (P7) — **P1**
**목적**: 사용자 "듣기 노동" 최소화
**도구**:
- `pyloudnorm` — LUFS/True Peak 측정
- `librosa` — pitch/timbre/energy/MFCC
- `essentia` — 장르/무드 classifier (사전 학습 모델)
- Claude 멀티모달 — 오디오 + 가사 스크립트 동시 보고 "일치도" 판정 (새 feature)

**출력**: `quality/{song_id}.json`
```json
{
  "lufs": -16.4,
  "true_peak": -1.2,
  "vocal_breathiness": 0.72,
  "genre_match": {"bossa_nova": 0.89, "jazz": 0.74},
  "concept_fit": 0.81,
  "wit_delivery": 0.65,
  "overall_score": 0.78
}
```

### ★ F-02. 텔레그램 봇 인라인 리뷰 (P8) — **P1**
**목적**: 노트북/폰에서 바로 픽 가능
**신규 핸들러**:
- `/review {batch_id}` → 6곡 버튼 카드 전송
- inline buttons: `pick` / `extend` / `reject` / `edit_lyrics`
- 픽 시 자동 P9 후처리 큐 진입

### ★ F-03. 3시안 Suno 병렬 생성 (P5) — **P1**
**목적**: end-to-end 시간 3× 단축
**구현**:
- `suno_client.py` → `asyncio` 전환 (현재 동기 블로킹)
- `undetected-chromedriver` 3 인스턴스 병렬 (Cloudflare 세션 공유 or 분리 판단)
- Context Manager로 리소스 해제 보장

### ★ F-04. 후처리 체인 파이프라인화 (P9) — **P1**
**목적**: 마스터링까지 한 번에
**도구**:
- `demucs` (Facebook) — 4-stem 분리 (vocals/bass/drums/other)
- `pedalboard` (Spotify) — Python 내 VST-like 체인
  - Vocal chain: HighPass → DeEsser → Compressor → Reverb(Plate) → EQ
  - Bass chain: LowPass → Compressor(slow)
  - Other chain: Saturator → Stereo Widener
- `matchering` — 레퍼런스 트랙 매칭 (Laufey 특정 곡 로컬 보관)
- `ffmpeg` loudnorm — -14 LUFS EBU R128

**통합**: `scripts/postprocess.py --song-id {sid} --reference laufey_from_the_start.wav`

### ★ F-05. Drive 서비스 계정 JSON 수정 (버그) — **P1**
**목적**: 현 파이프라인 Drive 업로드 실패 해결
**작업**:
- `.env` `GOOGLE_CREDENTIALS_PATH` 점검
- 누락 필드(`token_uri`, `client_email`) 포함된 정상 JSON 교체
- `credentials/service-account.json` 템플릿 + `.env.example` 반영

### F-06. 컨셉 설계 자동화 (P1) — P2
**목적**: 사용자 시드 1줄 → 풍부한 브리프
**구현**:
- Claude API + 취향 DB (`data/taste_profile.json`)
- 레퍼런스 5곡 + 장르/무드/템포/키/제목후보

### F-07. 가사 3시안 병렬 (P2) — P2
**목적**: A/B/C 변주 자동 생성
**구현**:
- `asyncio.gather(*[claude_generate(seed, angle=a) for a in ['A','B','C']])`
- 각 시안별 라임·음절 자체 검증

### F-08. 아트워크 AI (P10) — P2
**목적**: 앨범/트랙 일관성
**도구**:
- DALL-E 3 (OpenAI) 또는 SDXL 로컬 (ComfyUI API)
- Be:Analogue 스타일 프롬프트 템플릿
- 썸네일 3시안 → 사용자 픽

### F-09. 메타데이터 자동 생성 (P11) — P2
**목적**: 플랫폼별 맞춤 문구
**구현**:
- `scripts/generate_platform_meta.py`
- YouTube / Spotify / Instagram Reel / Apple Music
- 한국어+영어 양면

### F-10. YouTube OAuth 자동 갱신 (P12) — P1
**목적**: 4-1에서 겪은 토큰 만료 재발 방지
**구현**:
- `refresh_token` 자동 사용
- 만료 감지 → 텔레그램 알림 + 재인증 링크 버튼
- 7일 전 사전 경고

### F-11. 피드백 루프 (P13) — P3
**목적**: 취향 DB 축적
**구현**:
- cron: 48시간 후 자동 수집
- YouTube Data API v3 (views/likes/comments)
- Claude로 코멘트 감정·키워드 분석
- `data/taste_profile.json` 가중치 업데이트

## 4. 기술 스택 (라이브러리 + 근거)

| 용도 | 선택 | 근거 |
|------|------|------|
| 오디오 분석 | `pyloudnorm` | ITU BS.1770 표준, 경량 |
| 특성 추출 | `librosa` | 표준, 문서 풍부 |
| 장르 분류 | `essentia-tensorflow` | 사전 학습 모델 다수 |
| 스템 분리 | `demucs v4` | 오픈소스 최고 품질 |
| 오디오 체인 | `pedalboard` (Spotify) | VST 호환, Python 네이티브 |
| 마스터링 | `matchering` | 레퍼런스 매칭 자동 |
| 라우드니스 | `ffmpeg loudnorm` | EBU R128 표준 |
| 병렬 | `asyncio` + `aiohttp` | Suno API 비동기 전환 |
| 아트워크 | `openai` (DALL-E 3) | 품질 안정성 |
| YouTube | `google-api-python-client` | 공식 |
| 큐 | 기존 `bot.py` + SQLite | 이미 있음, 재사용 |

## 11. 구현 세션 리뷰 코멘트 (2026-04-23)

### [R-1] F-03 Suno 3탭 병렬 — 실제 가능성 중간 위험
**현재 `suno_client.py` 구조**: `undetected-chromedriver` 인스턴스 1개 + Clerk JWT 1개를 `SunoClient`가 보유. 세션은 계정 단위.
- **Turnstile 비용**: 프로필별 최초 1회만. user-data-dir 분리 시 각 탭마다 해결 필요 → **병렬 초기화 비용 3배**.
- **계정 세션 공유 시**: 같은 Clerk JWT로 동시 3요청이 Suno 백엔드에서 rate limit / "in-progress" 거부로 튕길 수 있음 (실측 전 가설).
- **권장**: asyncio 전환 **이전에** PoC 필수 — 단일 JWT·단일 크롬 프로필에서 `/generate` 3건 동시 POST 결과(HTTP 429 / 200 / 큐잉) 측정. 튕기면 탭 3개 + 프로필 3개로 분리.
- **폴백**: 순차 + 다운로드 병렬이 현실적. P5 직렬 9분 → P5(9분) + P6(20s) 정도의 단축도 의미 있음.

### [R-2] F-05 서비스 계정 JSON — 현재 파일은 **OAuth client_secret 포맷**
- 파일은 존재하지만 (`client_secrets.json`, 405B) 내용이 `{"installed": {...}}` 구조 = OAuth 사용자 플로우용.
- `drive_uploader.py`는 `service_account.Credentials.from_service_account_file`을 호출하므로 포맷 불일치로 실패.
- **수정 방향**: (권장) GCP 콘솔에서 서비스 계정 새로 발급 + `credentials/drive-sa.json`로 저장 + `GOOGLE_CREDENTIALS_PATH` 갱신. 아니면 `drive_uploader.py`를 OAuth 플로우로 재작성(YouTube의 `token.json` 구조 재사용). 전자가 운영 단순.

### [R-3] F-04 후처리 레퍼런스 매칭
- Matchering은 "하나의 레퍼런스" 강결합. 곡마다 레퍼런스가 다른 Music Lab에는 **장르/무드 기반 레퍼런스 라이브러리**가 필요.
- 최소 구성: `reference/{bossa_nova}.wav`, `{kpop_ballad}.wav`, `{lofi}.wav` 3~5개부터 시작. 사용자가 각자 Laufey/폴킴/이디오테잎 1곡씩 원본 보관.
- `postprocess.py --reference auto` 모드로 장르 태그에서 자동 선택하도록 설계 여지.

### [R-4] F-07 가사 3시안 병렬 — 비용 관측 필요
- Claude API `claude-opus-4-7` 가사 한 시안 = 입력 2~3k + 출력 1k 가정 시 시안당 약 $0.09 (Opus 기준). 3시안 × 10곡/주 = $2.7/주. 현 규모에선 미미.
- 단, 재치 스코어 검증/반복 재생성까지 들어가면 3~5배. 일간 비용 알림(리스크 테이블) 초기부터 필수.

### [R-5] 병렬 시간 추정 오차
- P5 Suno는 웹 UI 큐 대기 변동(2~8분)이 커서 "3분" 가정은 낙관적. **end-to-end 15분**은 베스트 케이스.
- 보수적 수치: Phase 1 완료 시 **첫 E2E 실측 후 추정 갱신** 필요.

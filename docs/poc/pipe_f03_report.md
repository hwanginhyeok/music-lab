**작성일:** 2026-05-05

## 1. 요약 (TL;DR)

**권장 안:** 병렬화 보류 + 프롬프트 최적화 + 자동 재시도 도입

**핵심 이유:**
1. 단일 JWT/Driver 인스턴스 공유는 race condition 초래 (고위험)
2. Suno 백엔드의 계정당 동시 생성 슬롯 제한으로 병렬화 효과 불확실
3. 45분(9곡)은 Suno SLA 내 정상이므로 ROI 부정적
4. 계정 정지(R-1) 리스크를 감수할 만큼 시급하지 않음

**대안:** 재생성 횟수를 줄이는 게 진짜 병목 해결 (프롬프트 품질)

## 3. 현황 분석

### 3.1 곡 생성 흐름

```
suno_pipeline.py
  ↓
SunoClient.generate()
  ↓
Chrome 자동화 (Selenium) + Clerk JWT 폴링
  ↓
한 번 Create로 2곡 동시 생성
  ↓
폴링: 5초 간격, 300초 타임아웃
  ↓
완료 시 mp3 다운로드 + 메타데이터 저장
```

**특징:**
- 단일 `self._driver` 인스턴스 (Chrome)
- 단일 `self._jwt` 인스턴스 변수
- 동시성 제어 없음 (lock 없음)

### 3.2 인증 핵심 (Clerk JWT)

```
1. SUNO_COOKIE (from .env)
   ↓
2. GET /v1/client?_clerk_js_version → session id 추출
   ↓
3. POST /sessions/{sid}/tokens → JWT 획득
   ↓
4. JWT로 Suno API 호출 (Bearer: {jwt})
   ↓
5. 폴링마다 JWT refresh
```

**설계상 취약점:**
- `self._jwt`는 인스턴스 변수 → 멀티 스레드에서 race condition
- JWT는 짧은 수명 (초단위) → 폴링 타이밍에 따라 충돌 가능

### 3.3 병렬화 시 위험 지점 (5단계 식별)

| 위험 등급 | 지점 | 영향 | 완화 난이도 |
|---------|------|------|-----------|
| **고위험** | `self._driver` 단일 인스턴스 공유 | 탭 전환 충돌, element not found | 높음 (멀티 드라이버 필요) |
| **고위험** | `self._jwt` 인스턴스 변수 race condition | JWT 덮어쓰기 → 401 Unauthorized | 중간 (lock 또는 로컬 변수화) |
| **중위험** | 기존 곡 ID 스냅샷 비교 | 타 요청의 곡까지 인식 → 완료 오판 | 낮음 (request ID 기반 필터링) |
| **저위험** | 폴링 간격 (5초) | Cloudflare 429 가능성 (미확인) | 낮음 (간격 조정) |

## 4. Suno Rate Limit 알려진 정보

| 항목 | 값 | 신뢰도 | 출처 |
|------|-----|--------|------|
| 공식 API | 없음 (비공개 엔드포인트만) | 높음 | Suno 공식 문서 |
| 공식 rate limit 명시 | 없음 | 높음 | Suno ToS |
| Pro 플랜 동시 슬롯 | 2-3개 추정 | 중간 | 커뮤니티 경험 |
| Premier 플랜 동시 슬롯 | 3-5개 추정 | 중간 | 커뮤니티 경험 |
| 30초 미만 간격 | Cloudflare 차단 | 높음 | 다수 사용자 보고 |
| 5초 미만 폴링 | 429 가능성 (미확인) | 낮음 | 추정 |
| 계정당 일일 생성 한도 | 명시 없음 | 중간 | 미확인 |

**미확인 영역:**
- 정확한 슬롯 수 (Pro/Premier)
- 일일/월간 생성 상한
- JWT 단위 또는 계정 단위 제한 여부
- IP ban 패턴

## 5. 3가지 PoC 안 비교

| 안 | 방식 | 난이도 | 가속도 | 리스크 | ToS | 추천 |
|----|------|--------|--------|--------|-----|------|
| **안 1** | 단일 JWT × asyncio | 3/5 | 1.5x | 매우 높음 | 위반 | ✗ |
| **안 2** | 단일 Driver 멀티 탭 | 2/5 | 1.2x | 높음 | 회색 | △ |
| **안 3** | 멀티 프로필 (계정 N개) | 4/5 | ~Nx | 중간 | OK | △ |
| **안 0** | 병렬화 안 함 + 최적화 | 1/5 | 1.0x | 낮음 | OK | ✓ |

### 안 1: 단일 JWT × asyncio

- **방식:** `asyncio.create_task()`로 동시 N요청, `self._jwt` 공유
- **장점:** 코드 변경 최소, asyncio.gather로 간단 구현
- **단점:** 
  - JWT race condition (고위험)
  - Driver 단일 인스턴스 → 탭 충돌
  - Suno ToS 위반 가능성 (자동화 병렬 요청)
- **예상 문제:** 타 요청의 JWT로 요청 → 401 → 전체 실패

### 안 2: 단일 Driver 멀티 탭

- **방식:** `driver.new_tab()`으로 탭 2개 열어서 병렬 생성
- **장점:** 단일 Driver라 리소스 적음, 탭 간섭 최소화 가능
- **단점:**
  - JWT 변수 공유 문제 (lock 필요)
  - Suno 백엔드가 계정당 슬롯 제한이면 무의미
  - ToS 회색 지대 (브라우저 자동화)
- **예상 효과:** 슬롯이 2개 이상이면 1.2x, 아니면 1.0x

### 안 3: 멀티 프로필 (계정 N개)

- **방식:** Chrome 프로필 N개 → Suno 계정 N개 → 완전 병렬
- **장점:** 계정별 슬롯 독립 → 이론적 N배 가속
- **단점:**
  - 비용 (Pro $10/월 × N)
  - 운영 오버헤드 (VNC N개, 캡차 N개)
  - JWT/Driver 인스턴스 N개 관리
- **예상 효과:** 월 EP 3개 이상 시 검토

### 안 0: 병렬화 보류 + 최적화

- **방식:** 시리얼 생성 유지 + 프롬프트 최적화 + 자동 재시도
- **장점:** 리스크 최소, ToS 준수, 즉시 실행 가능
- **단점:** 절대 시간 단축 없음
- **예상 효과:** 재생성률 30% → 10%로 감소 시 실제 소요 시간 ~30분

## 6. 비판적 검토: 병렬화가 진짜 필요한가?

### 6.1 45분은 Suno SLA 내 정상
9곡 × 5분 = 45분 (이론값, 실측 동일). 백엔드 생성 시간은 병렬화로 단축 불가.

### 6.2 병렬화로 줄이는 것은 우리 측 대기 시간만
| 구분 | 시리얼 | 병렬(2x) | 감소 |
|------|--------|---------|------|
| Suno 백엔드 | 45분 | 45분 | 0분 |
| 우리 측 폴링/대기 | 45분 | ~23분 | 22분 |

### 6.3 YAGNI
월 1-2개 EP면 생성 빈도 낮음. 45분 대기는 밤새 돌면 해결, 급한 일정 없음.

### 6.4 안 1/2는 백엔드 슬롯 제한이면 무의미
Suno 계정당 슬롯 2개 제한 시: 안 1/2 모두 실제 1.0x. 효과 불확실, 리스크 확실.

### 6.5 안 3은 비용 대비 효과 의문
Pro $10/월 × 2 = $20/월. 월 2개 EP → 1회당 $10. 22분 단축 비용으로 과다, 운영 오버헤드도 큼.

## 7. 권장 안: 안 0 (병렬화 보류) + 최적화

### 7.1 결정

**✓ 안 0 채택:** 병렬화 보류 + 프롬프트 최적화 + 자동 재시료

### 7.2 이유

1. **ROI 부정적:** 안 1/2는 슬롯 제한 미확인으로 효과 불확실, 안 3은 $20/월+운영오버헤드
2. **진짜 병목은 재생성:** 9곡 중 ~3곡 재생성(33%). 원인은 프롬프트 모호함 → 템플릿 구체화로 해결
3. **R-1 리스크 불필요:** 계정 정지 시 복구 불가, 45분 대기가 계정 소실보다 낮음
4. **YAGNI 준수:** 월 1-2개 EP면 시리얼로 충분, 급한 일정 없음

### 7.3 대안 구현

- **프롬프트 템플릿:** 악기/템포/분위기 명시 + 참고 곡 링크
- **자동 재시도:** 타임아웃 시 1회 재시도, 실패 시 로그 후 다음 곡
- **진행률 시각화:** 텔레그램 "3/9 완료" 알림으로 대기 불안감 해소

### 7.4 향후 병렬화가 정말 필요하면

**우선순위:**
1. **안 2 (멀티 탭) PoC 1일**
   - 사용자 승인 후 1탭 + 2탭 시간 비교 1회만
   - 실제 병렬되는지만 확인
   - 슬롯 제한 확인

2. **안 3 (멀티 프로필)**
   - 월 EP 3개+ 시점에 검토
   - 비용/효과 재검토
